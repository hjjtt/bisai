# 基于大模型的实训成果智能核查与评价系统方案

## 一、项目背景

本项目面向第十五届“中国软件杯”信息技术应用创新“智赋百业”专项赛 B1 赛题，围绕实训成果上传、智能解析、自动核查、多维评价和报表导出等核心要求，设计并实现一套基于 B/S 架构的实训成果智能核查与评价系统。

系统部署运行在龙架构 LoongArch 与麒麟高级服务器操作系统环境中，结合大模型、OCR、多模态理解、RAG 检索增强、规则引擎和可视化报表技术，帮助教师高效完成实训成果检查、评分和反馈工作。

考虑到比赛环境可能存在国产 CPU 架构、操作系统版本、模型推理框架和第三方组件兼容性差异，系统采用“业务系统本地部署、AI 能力可插拔接入”的架构设计。大模型、OCR、Embedding、Rerank 等能力既可以本地部署，也可以通过 ModelScope 或其他兼容 API 服务调用，确保在 LoongArch + 麒麟环境下具备可落地性和应急替代能力。

## 二、建设目标

系统建设目标如下：

1. 支持学生上传 Word、PDF、图片等多种格式的实训成果文件。
2. 结合 OCR、多模态大模型和文档解析技术自动读取成果内容，提取实训成果中的核心信息。
3. 自动核查实训要求与实训效果是否匹配，识别实训步骤完整性。
4. 自动发现成果中的逻辑漏洞、内容缺失和前后矛盾问题。
5. 支持教师自定义评价指标、权重和评分规则。
6. 结合大模型生成客观评分、主观评价建议和改进意见。
7. 支持生成实训评价报告、班级统计报表和可视化图表。
8. 支持 Word、PDF、Excel 等格式导出。
9. 满足 LoongArch 与麒麟服务器操作系统部署要求，并提供不兼容场景下的 API 化兜底方案。

## 三、系统总体设计

### 3.1 总体架构

系统采用前后端分离的 B/S 架构，主要由前端展示层、后端业务层、智能分析层、数据存储层和部署运行层组成。

```text
浏览器端
  ↓
前端应用 Vue 3
  ↓
后端服务 Spring Boot
  ↓
业务服务层
  ├── 用户权限服务
  ├── 实训任务服务
  ├── 文件管理服务
  ├── 文档解析服务
  ├── 智能核查服务
  ├── 智能评分服务
  ├── RAG 检索服务
  ├── 模型适配服务
  └── 报表生成服务
  ↓
数据库 / 文件存储 / 向量库 / 缓存
  ↓
模型服务层
  ├── Spring AI 编排
  ├── ModelScope 多模态 API
  ├── OCR 服务
  └── Embedding / Rerank 服务
  ↓
LoongArch + 麒麟高级服务器操作系统
```

### 3.2 技术路线

推荐技术选型如下：

| 层级 | 推荐技术 | 说明 |
|---|---|---|
| 前端 | Vue 3 + TypeScript + Element Plus + Pinia + Vue Router + Axios | 技术路线明确，适合管理系统快速开发 |
| 后端 | Spring Boot | 企业级应用成熟，适合国产化部署 |
| 大模型编排 | Spring AI + 自定义 Adapter | 统一封装 Chat、Embedding、Prompt、RAG 和工具调用 |
| 数据库 | openGauss / PostgreSQL / 达梦数据库 | 根据比赛环境选择 |
| 缓存 | Caffeine 本地缓存，条件允许时升级 Redis | MVP 阶段降低 LoongArch 部署风险 |
| 文件存储 | MinIO / 本地文件系统 | 存储上传文件和生成报告 |
| 文档解析 | Apache POI、PDFBox、OCR 服务、多模态 API | 解析 Word、PDF、图片、截图和表格 |
| 大模型 | ModelScope API / 本地模型服务 | 负责内容抽取、核查、评分和反馈生成 |
| RAG 检索 | 向量检索 + Rerank + 可选 LazyGraphRAG | 支持实训要求、评分标准、知识库和历史样例检索 |
| 向量数据库 | PostgreSQL + pgvector | 与业务库统一部署，降低 LoongArch 环境适配风险 |
| 可视化 | ECharts | 展示成绩分布和统计图表 |
| 报表导出 | EasyExcel、Freemarker、PDF 工具 | 导出 Excel、Word、PDF |
| 文件预览 | pdf.js + Word 转 PDF | 支持教师在线查看原始成果 |

### 3.3 大模型与 RAG 接入架构

系统后端优先采用 Spring Boot 技术栈，可引入 Spring AI 作为大模型应用编排框架，用于统一管理模型调用、Prompt 模板、Embedding、向量检索和 RAG 流程。对于 ModelScope、通义千问或其他兼容 OpenAI API 格式的模型服务，可通过 Spring AI 的 ChatModel、EmbeddingModel 和 VectorStore 接口接入；如果接口格式不完全兼容，则通过自定义 `ModelScopeClient` 和模型适配器封装。

```text
业务服务
  ↓
AI 编排层 Spring AI
  ├── Prompt 模板管理
  ├── ChatModel 调用
  ├── EmbeddingModel 调用
  ├── VectorStore 检索
  ├── Tool Calling
  └── 对话上下文管理
  ↓
模型适配层
  ├── ModelScopeClient
  ├── OcrClient
  ├── RerankClient
  └── LocalModelClient
  ↓
模型与算法服务
```

该设计避免系统与单一模型厂商强绑定。当 ModelScope 多模态接口、OCR 服务或本地模型不可用时，可以通过更换适配器切换到其他模型服务，业务代码不需要大规模改造。

### 3.4 智能核查技术策略

系统采用“规则核查 + RAG 检索 + 大模型判断 + 教师复核”的组合策略。

| 场景 | 推荐策略 | 说明 |
|---|---|---|
| 文件格式、提交时间、章节是否存在 | 规则引擎 | 结果稳定，便于自动扣分 |
| 实训要求与成果内容匹配 | 向量检索 + Rerank | 检索相关段落后再交给模型判断 |
| 跨章节、跨文件、多步骤关系判断 | 可选 LazyGraphRAG | 适合复杂关系、指标关联和多跳推理 |
| 截图、图表、界面内容理解 | 多模态大模型 | 处理传统文本解析难以覆盖的信息 |
| 评分建议和改进意见 | 大模型生成 | 输出自然语言解释，供教师复核 |

LazyGraphRAG 不作为系统必须依赖，而作为复杂检索增强能力。当实训报告、评分标准、历史优秀样例和课程知识点之间存在较多关联时，可采用 LazyGraphRAG 或类似轻量级 GraphRAG 方案提升跨文档匹配能力；对于普通单文档问答和完整性检查，优先使用向量检索、关键词规则和 Rerank，以降低实现复杂度和调用成本。

### 3.5 向量数据库选型

系统推荐使用 PostgreSQL + pgvector 作为默认向量检索方案。原因如下：

- 与业务数据库统一部署，减少 Milvus、Chroma 等独立组件带来的国产化适配风险。
- PostgreSQL 和 openGauss 生态在国产服务器环境中更容易验证和替换。
- 比赛 MVP 阶段的数据规模有限，pgvector 足以支撑实训报告、评分标准、知识库和历史样例的检索。
- 后续数据规模增大时，可将向量检索层迁移到 Milvus 等独立向量数据库，业务层通过 VectorStore 接口保持不变。

Embedding 模型维度需要与 pgvector 字段定义保持一致。MVP 阶段建议选择固定维度的文本 Embedding 模型，例如 1024 维向量，并在建表时使用 `vector(1024)`；如果后续更换模型，应同步调整向量字段维度或重建向量索引。

向量检索默认流程如下：

```text
文档切片
  ↓
Embedding 生成
  ↓
写入 pgvector
  ↓
相似度召回 TopK
  ↓
Rerank 重排
  ↓
大模型核查或评分
```

### 3.6 异步任务处理设计

文档解析、OCR、多模态理解、Embedding、Rerank 和大模型评分均属于耗时操作，系统采用异步任务机制处理，避免教师或学生页面长时间阻塞。

MVP 阶段推荐使用数据库任务表 + 定时轮询方式，降低部署复杂度；如果现场环境允许，可升级为 Redis Stream。

```text
提交成果
  ↓
创建异步任务
  ↓
任务进入 PENDING
  ↓
后台 Worker 拉取任务
  ↓
状态变为 RUNNING
  ↓
执行解析、OCR、模型调用
  ↓
成功 SUCCESS / 失败 FAILED
```

任务状态包括：

| 状态 | 含义 |
|---|---|
| PENDING | 等待执行 |
| RUNNING | 正在执行 |
| SUCCESS | 执行成功 |
| FAILED | 执行失败 |
| RETRYING | 等待重试 |
| CANCELED | 已取消 |

失败重试策略：

- 网络超时、API 限流、临时服务不可用等错误允许自动重试。
- 文件损坏、格式不支持、权限不足等错误不自动重试，直接提示教师或学生处理。
- 单个任务最多重试 3 次，采用递增等待时间。
- 重试失败后保留错误日志，允许教师手动重新发起智能评价。

### 3.7 文档切片与成本控制

文档切片采用“章节优先、段落辅助、长度兜底”的策略：

- 优先按标题层级、目录结构和页码进行切分。
- 无明显标题时按段落切分。
- 单个切片建议控制在 500-800 中文字符。
- 相邻切片保留 80-120 字重叠，避免上下文断裂。
- 表格、图片说明和截图识别结果作为独立切片，并保留来源页码。
- 每个切片记录来源文件、页码、章节、解析方式和原文证据。

模型调用成本控制机制：

- 按任务、班级、教师设置每日调用次数和 token 上限。
- 对同一文件内容计算哈希，重复提交时复用解析和 Embedding 结果。
- 优先使用规则核查和检索结果过滤明显问题，只对必要片段调用大模型。
- 对模型调用记录 input_tokens、output_tokens、模型名称和任务类型。
- 达到配额阈值时给管理员预警，并自动降级为“规则核查 + 教师人工复核”模式。

### 3.8 缓存设计

MVP 阶段推荐使用 Caffeine 本地缓存，减少外部中间件依赖，降低 LoongArch + 麒麟环境部署不确定性。缓存主要用于读取频繁、变化较少的数据。

缓存内容包括：

- 用户权限和菜单配置
- 系统配置项
- 评价模板和指标配置
- Prompt 模板
- 模型服务配置
- 文件解析结果摘要

当系统部署环境确认 Redis 可用后，可将缓存升级为 Redis，进一步支持多实例部署、异步任务队列和分布式限流。业务代码通过缓存接口封装，避免直接绑定某一种缓存实现。

## 四、用户角色设计

### 4.1 学生角色

学生主要负责查看实训任务、上传实训成果和查看评价结果。

主要功能：

- 登录系统
- 查看课程和实训任务
- 上传 Word、PDF、图片等成果文件
- 查看提交状态
- 查看教师发布的评价结果
- 下载个人评价报告

### 4.2 教师角色

教师主要负责创建任务、配置评价标准、查看智能核查结果并确认最终评分。

主要功能：

- 创建课程
- 创建实训任务
- 设置提交时间和文件要求
- 配置评价指标和权重
- 查看学生提交情况
- 启动智能解析和智能核查
- 查看系统评分建议
- 人工复核并调整评分
- 生成和导出评价报告

### 4.3 管理员角色

管理员负责系统基础数据、权限和运行配置维护。

主要功能：

- 用户管理
- 班级管理
- 课程管理
- 角色权限管理
- 模型服务配置
- 系统日志查看
- 数据备份与恢复

## 五、核心功能设计

### 5.1 实训成果上传与解析

#### 5.1.1 文件上传

系统支持学生上传以下类型文件：

- Word 文档：`.doc`、`.docx`
- PDF 文档：`.pdf`
- 图片文件：`.jpg`、`.jpeg`、`.png`
- 表格文件：`.xls`、`.xlsx`
- 压缩包文件：`.zip`

上传控制规则：

- 限制文件类型
- 限制文件大小
- 限制提交截止时间
- 支持重新提交
- 保留历史提交版本
- 支持批量上传
- 支持教师查看原始文件

#### 5.1.2 文档解析

系统根据不同文件类型采用不同解析策略。

| 文件类型 | 解析内容 |
|---|---|
| Word | 标题、正文、表格、图片说明 |
| PDF | 文本、段落、页码、图文内容 |
| 图片 | OCR 文字、界面元素、图表文字 |
| Excel | 表格数据、统计结果、指标数据 |
| ZIP | 解压后识别代码、文档、图片和配置文件 |

解析后的内容统一转换为结构化文本，保存到数据库中，用于后续智能核查和评分。

#### 5.1.3 OCR 与多模态协同解析

系统不将 OCR 与多模态大模型设计为互相替代关系，而是采用协同解析方式：

| 能力 | 主要作用 | 使用场景 |
|---|---|---|
| OCR | 将图片、扫描件、截图中的文字转为可检索文本 | 建库、检索、证据定位、字段抽取 |
| 文档解析 | 从 Word、PDF、Excel 中提取正文、表格和结构信息 | 文本类成果、报告类成果 |
| 多模态大模型 | 理解图片、界面截图、图表、复杂版式和低质量扫描件 | OCR 置信度低、图文关系复杂、截图需要语义判断 |
| 大模型文本抽取 | 从解析文本中抽取实训目标、功能、测试、总结等结构化字段 | 智能核查和评分 |

OCR 仍然保留的原因如下：

- RAG 检索和向量建库需要稳定的文本输入。
- 教师复核时需要看到可追溯的原文、页码和截图位置。
- OCR 批量处理成本低，适合对大量文档进行初筛。
- 多模态大模型适合处理疑难内容，不适合替代全部基础解析任务。

因此，系统采用“先文档解析和 OCR，后多模态补充判断”的流程。对于清晰文本，直接使用解析结果；对于截图、图表、扫描件或 OCR 置信度较低的页面，再调用 ModelScope 多模态 API 进行二次理解。

#### 5.1.4 大模型信息抽取

系统调用大模型从成果内容中抽取核心信息：

- 实训项目名称
- 实训目标
- 需求分析
- 系统设计
- 数据库设计
- 功能实现
- 测试过程
- 运行截图说明
- 实训总结
- 创新点
- 存在问题
- 改进建议

抽取结果以 JSON 结构保存，便于系统展示和算法评分。

### 5.2 智能核查功能

智能核查功能用于判断学生成果是否满足教师设置的实训要求。

#### 5.2.1 实训要求匹配

教师创建任务时配置成果要求，例如：

- 是否包含需求分析
- 是否包含系统设计
- 是否包含数据库设计
- 是否包含核心功能实现说明
- 是否包含测试用例
- 是否包含运行截图
- 是否包含实训总结
- 是否提交源代码或项目包

系统将学生成果与实训要求逐项匹配，输出核查结果。

示例：

| 检查项 | 检查结果 | 说明 |
|---|---|---|
| 需求分析 | 已完成 | 包含功能需求和非功能需求 |
| 数据库设计 | 部分完成 | 有表结构，但缺少字段说明 |
| 测试结果 | 未完成 | 未发现测试用例和测试截图 |
| 运行截图 | 已完成 | 检测到多个系统界面截图 |

#### 5.2.2 实训步骤完整性核查

系统按照完整实训流程进行检查：

```text
任务理解
  ↓
需求分析
  ↓
系统设计
  ↓
功能实现
  ↓
测试验证
  ↓
总结反思
```

核查内容包括：

- 是否缺少关键步骤
- 是否存在章节标题但正文为空
- 是否存在内容顺序混乱
- 是否缺少关键图表
- 是否缺少运行结果证明
- 是否缺少测试验证过程
- 是否缺少实训总结和改进意见

#### 5.2.3 逻辑漏洞识别

系统结合规则引擎和大模型识别以下问题：

- 需求与实现不一致
- 数据库设计无法支撑功能描述
- 功能截图与文字说明不匹配
- 测试用例未覆盖核心功能
- 系统架构说明前后矛盾
- 实验结果与总结结论不一致
- 报告中存在明显复制粘贴痕迹
- 内容过于空泛，缺少实际成果证明

#### 5.2.4 风险等级划分

核查结果按照风险等级展示。

| 风险等级 | 含义 | 处理建议 |
|---|---|---|
| 高风险 | 关键内容缺失，严重影响评分 | 教师重点复核 |
| 中风险 | 部分内容不完整或存在疑点 | 教师人工确认 |
| 低风险 | 格式、表述或轻微问题 | 可作为反馈建议 |

### 5.3 多维度评价功能

#### 5.3.1 评价指标体系

系统支持教师自定义评价指标、分值和权重。

示例指标体系：

| 一级指标 | 二级指标 | 权重 |
|---|---|---|
| 文档规范性 | 结构完整、格式规范、表达清晰 | 15% |
| 需求分析 | 需求完整性、合理性、可实现性 | 20% |
| 系统设计 | 架构设计、数据库设计、模块划分 | 20% |
| 功能实现 | 功能完成度、运行效果、截图证明 | 25% |
| 测试验证 | 测试用例、测试结果、问题修复 | 10% |
| 创新总结 | 创新点、反思深度、改进建议 | 10% |

#### 5.3.2 客观评分

客观评分基于明确规则自动计算。

评分依据包括：

- 是否按时提交
- 是否提交指定文件
- 是否包含必要章节
- 是否包含运行截图
- 是否包含测试结果
- 字数是否达标
- 文件格式是否符合要求

#### 5.3.3 智能主观评分

主观评分由大模型根据评价标准生成建议分数和评分理由。

评价内容包括：

- 需求分析质量
- 设计方案合理性
- 功能实现完整度
- 测试验证充分性
- 文档表达清晰度
- 总结反思深度
- 创新性和应用价值

#### 5.3.4 教师最终评分

系统评分仅作为辅助，最终成绩由教师确认。

教师可执行以下操作：

- 查看系统建议分
- 查看评分理由
- 修改单项得分
- 填写教师评语
- 发布最终成绩
- 导出评价报告

#### 5.3.5 评分可解释性

每个评分项必须展示评分依据，避免只给分不解释。

示例：

```text
功能实现：21 / 25 分

评分依据：
报告中描述了用户登录、数据管理、结果展示等核心功能，并提供了主要界面截图。
但缺少异常处理和权限控制说明，因此扣 4 分。
```

### 5.5 知识库管理功能

RAG 检索需要稳定的知识来源，系统提供知识库管理功能，用于维护实训指导书、评分标准、课程知识点、历史优秀样例和常见问题库。

主要功能：

- 创建课程知识库
- 上传实训指导书和评分标准
- 上传历史优秀成果样例
- 对知识库文档进行解析、切片和向量化
- 设置知识库适用课程和任务范围
- 启用或停用某个知识库
- 查看知识库文档的解析状态和向量化状态

知识库内容不会直接覆盖教师评分规则，而是作为智能核查和评分建议的参考依据。最终评分仍以当前任务绑定的评价模板和教师确认结果为准。

### 5.6 批量评价与进度管理

教师可对一个任务下的多个学生提交记录批量启动智能评价。系统通过异步任务队列控制并发，避免同一时间大量调用 OCR 和大模型 API。

批量处理功能包括：

- 批量启动解析
- 批量启动智能核查
- 批量生成评分建议
- 批量导出个人报告
- 批量导出班级统计报表
- 查看批量任务进度
- 查看失败任务并支持重新执行

并发控制规则：

- 单个教师同一时间只允许发起有限数量的批量任务。
- 同一提交记录不能重复进入同类任务队列。
- 大模型 API 达到限流或配额上限时自动暂停队列。
- 已失败任务允许手动重试，不影响其他任务结果。

### 5.7 评分一致性与校准机制

为了保证同一任务下不同学生的评分尺度一致，系统提供评分校准机制。

主要设计：

- 同一实训任务绑定固定评价模板，评分过程中不得随意变更。
- 模型评分时统一使用任务要求、评价指标、评分规则和相同 Prompt 版本。
- 教师可抽取若干学生成果作为校准样本，先人工确认评分标准。
- 系统根据校准样本生成评分参考，后续评分保持相同尺度。
- 教师调整分数时记录调整原因，便于复盘评分偏差。
- 班级评分完成后提供各指标分布图，辅助发现异常高分或异常低分。

### 5.8 文件在线预览功能

教师复核时需要频繁查看学生原始成果文件，系统提供在线预览能力，减少下载文件和本地打开的操作成本。

预览方案：

| 文件类型 | 预览方式 |
|---|---|
| PDF | 使用 pdf.js 在浏览器中直接预览 |
| Word | 后端转换为 PDF 后预览，原文件仍保留下载 |
| 图片 | 浏览器直接预览，支持缩放 |
| Excel | 解析为表格 HTML 或转换为 PDF 预览 |
| ZIP | 展示压缩包目录树，允许教师查看解压后的安全文件列表 |

为保证安全，在线预览接口必须进行权限校验，学生只能查看自己的文件，教师只能查看其课程任务下的学生文件，管理员按权限查看系统文件。

## 六、关键业务流程

### 6.1 教师创建任务流程

```text
教师登录
  ↓
创建课程
  ↓
创建实训任务
  ↓
设置提交时间
  ↓
设置成果文件要求
  ↓
配置实训要求
  ↓
选择评价模板
  ↓
发布任务
```

### 6.2 学生提交成果流程

```text
学生登录
  ↓
进入实训任务
  ↓
上传成果文件
  ↓
系统校验文件格式
  ↓
保存原始文件
  ↓
生成提交记录
  ↓
进入待解析状态
```

### 6.3 智能评价流程

```text
读取提交文件
  ↓
文档解析
  ↓
OCR 识别 / 多模态理解
  ↓
结构化信息抽取
  ↓
实训要求与评分标准检索
  ↓
向量召回 / Rerank / 可选 LazyGraphRAG
  ↓
匹配实训要求
  ↓
完整性核查
  ↓
逻辑漏洞识别
  ↓
多维度评分
  ↓
生成评分理由
  ↓
生成初评报告
```

智能评价过程中，系统会将教师配置的评分标准、任务要求、课程知识点和学生成果内容一起纳入上下文。对于篇幅较长的成果文件，系统不会一次性将全文提交给大模型，而是先进行切片、Embedding、向量召回和 Rerank，再将最相关片段交给模型判断。这样可以降低 token 成本，并提升评分依据的可解释性。

如果某一核查项涉及跨章节或跨文件关系，例如“需求分析中的功能是否在设计和截图中都有体现”，系统可启用 LazyGraphRAG 思路，将需求、功能模块、截图说明、测试用例等节点建立临时关系图，用于辅助多跳匹配。

### 6.4 教师复核流程

```text
教师查看系统评分
  ↓
查看核查问题
  ↓
查看原始成果文件
  ↓
调整单项分数
  ↓
填写教师评语
  ↓
确认最终成绩
  ↓
发布评价结果
```

### 6.5 异步任务状态流转

```text
创建任务
  ↓
PENDING 等待执行
  ↓
RUNNING 执行中
  ↓
SUCCESS 成功
```

异常情况下进入以下分支：

```text
RUNNING 执行中
  ↓
FAILED 失败
  ↓
RETRYING 等待重试
  ↓
RUNNING 重新执行
```

如果重试次数超过上限，任务保持 FAILED 状态，并在教师端展示失败原因。教师可以选择重新执行，也可以进入人工评分模式。

### 6.6 纯人工评分降级流程

当外部模型 API 不可用、本地模型无法部署或网络中断时，系统保留完整的人工评分闭环。

```text
教师查看学生提交文件
  ↓
查看基础解析结果
  ↓
按评价模板逐项打分
  ↓
填写评语
  ↓
确认最终成绩
  ↓
导出评价报告
```

纯人工评分模式下，系统仍提供文件管理、评价模板、分数计算、报表导出和操作留痕能力，只暂停智能解析、智能核查和智能评分建议。

## 八、大模型提示词设计

提示词模板由 Spring AI 或系统自定义 Prompt 管理模块统一维护。所有模型输出优先要求 JSON 结构，业务层对返回结果进行格式校验、字段补全和异常重试，避免直接信任模型自由文本。

### 8.1 内容抽取提示词

```text

需要抽取：
1. 实训项目名称
2. 实训目标
3. 需求分析

5. 功能实现
6. 数据库设计
7. 测试过程
8. 实训总结
9. 创新点
10. 存在问题

请返回 JSON：
{
  "project_name": "实训项目名称",
  "training_goal": "实训目标",
```

## 九、评分算法设计

### 9.1 综合评分公式

```text
最终得分 = Σ（指标得分 × 指标权重）
```

系统评分由客观评分、主观评分和教师复核三部分组成：

| 类型 | 来源 | 说明 |
|---|---|---|
| 客观评分 | 规则引擎 | 按时提交、文件格式、必要章节、截图、测试结果、字数等明确规则 |
| 主观评分 | 大模型建议 | 需求分析质量、设计合理性、功能完成度、测试充分性、总结反思等 |
| 教师复核 | 教师确认 | 教师可调整系统建议分，并填写调整原因 |

单项指标得分计算方式：

```text
系统建议分 = 客观基础分 + 主观建议分 - 风险扣分
最终指标得分 = 教师确认分
```

如果教师未调整分数，则默认采用系统建议分作为最终指标得分。

分数边界规则：

```text
0 ≤ 单项指标得分 ≤ 单项指标满分
0 ≤ 最终得分 ≤ 100
```

当客观评分和主观评分分开展示时，系统应同时展示客观命中项、主观评分理由和扣分依据，便于教师判断是否采纳。

示例：

```text
总分 = 文档规范性 × 15%
     + 需求分析 × 20%
     + 系统设计 × 20%
     + 功能实现 × 25%
     + 测试验证 × 10%
     + 创新总结 × 10%
```

### 9.2 完整性评分

```text
完整性得分 = 已完成项数量 / 要求项总数量 × 100
```

### 9.3 风险扣分机制

| 风险等级 | 建议扣分 |
|---|---|
| 高风险 | 5-15 分 |
| 中风险 | 2-5 分 |
| 低风险 | 0-2 分 |

同一指标下多个风险项可以累加扣分，但风险扣分总计不超过该指标满分的 50%。扣分后仍需执行分数边界规则，确保单项指标得分不低于 0 分。跨指标风险应归属到对应指标分别计算，避免同一问题重复扣分。

### 9.4 教师复核优先级

系统按照以下顺序推荐教师优先复核：

```text
高风险成果
  ↓
低分成果
  ↓
缺失严重成果
  ↓
正常成果
```

## 十、国产化适配方案

系统需要适配以下环境：

- CPU 架构：LoongArch
- 操作系统：麒麟高级服务器操作系统
- 部署方式：服务器本地部署
- 访问方式：浏览器访问

LoongArch 与 x86_64、ARM64 属于不同 CPU 指令集架构，不能默认运行 x86_64 或 ARM64 二进制程序。因此，系统在选型时必须确认 JDK、数据库、OCR、文件解析库、容器镜像和模型推理框架是否提供 LoongArch 或麒麟系统可用版本。对于无法确认兼容性的组件，优先采用远程 API 或独立服务调用，避免核心业务系统部署失败。

### 10.1 适配重点

| 项目 | 适配要求 |
|---|---|
| JDK | 使用支持 LoongArch 的 OpenJDK |
| 数据库 | 选择支持国产系统的数据库 |
| OCR | 优先选择可本地部署方案，同时准备 API 调用兜底 |
| 文件解析 | 避免依赖仅支持 x86 的组件 |
| 容器 | 如使用 Docker 或 Podman，需确认 LoongArch 镜像支持 |
| 大模型 | 优先采用远程 API 或独立模型服务，本地模型作为增强方案 |
| Python 依赖 | 检查是否存在 LoongArch 可用 wheel，必要时源码编译 |
| 前端构建 | 建议在开发机完成构建，服务器只部署静态资源 |
| 第三方 SDK | 避免使用仅发布 x86_64 二进制包的闭源 SDK |

### 10.2 部署方案一：传统部署

```text
Nginx
  ↓
前端静态资源
  ↓
Spring Boot 后端服务
  ↓
openGauss / PostgreSQL
  ↓
MinIO
  ↓
本地模型服务 / 远程模型 API
```

### 10.3 部署方案二：容器化部署

```text
Docker / Podman
  ├── nginx
  ├── backend
  ├── database
  ├── minio
  └── model-service
```

比赛现场环境不确定时，建议优先准备传统部署方案，降低兼容风险。

### 10.4 AI 能力兼容性兜底方案

如果 LoongArch + 麒麟环境无法稳定运行本地模型、OCR SDK 或向量数据库组件，系统采用以下兜底策略：

| 风险场景 | 兜底方案 |
|---|---|
| 本地大模型推理框架不兼容 | 改为调用 ModelScope 或其他云端模型 API |
| 多模态模型无法本地部署 | 图片和截图通过多模态 API 分析 |
| OCR SDK 不支持 LoongArch | 使用 OCR HTTP 服务，或部署到 x86/ARM 旁路节点 |
| Python 依赖无法安装 | 将相关能力拆分为独立服务，主系统通过 REST 调用 |
| 容器镜像不支持 LoongArch | 改用传统部署，或重新构建 LoongArch 镜像 |
| 向量数据库服务不兼容 | 使用数据库全文检索 + 轻量向量服务替代 |
| API 不可用且本地模型无法部署 | 降级为纯人工评分模式，保留上传、查看、打分和导出能力 |

推荐部署形态如下：

```text
LoongArch + 麒麟服务器
  ├── Nginx
  ├── Spring Boot 业务系统
  ├── 数据库
  ├── 文件存储
  └── AI Adapter
        ↓ HTTP / HTTPS
      ModelScope API / OCR 服务 / 旁路模型服务器
```

这种方式可以保证比赛现场即使无法完成本地 AI 推理，也能优先通过 API 化能力完成文档解析、智能核查和评分演示；如果 API 也不可用，则降级为纯人工评分模式，保证核心业务流程不中断。

### 10.5 部署前验证清单

部署到正式环境前，应完成以下验证：

- 确认操作系统版本、内核版本和 CPU 架构。
- 确认 JDK、Maven、Node.js、Nginx 可正常运行。
- 确认数据库和文件存储可正常启动。
- 确认 Word、PDF、图片、Excel 解析流程可用。
- 确认 OCR 服务或 OCR API 可用。
- 确认 ModelScope 多模态 API、文本模型 API、Embedding API 调用正常。
- 确认向量检索、Rerank 和评分流程可正常返回结果。
- 确认导出 Word、PDF、Excel 文件无乱码。
- 确认教师、学生、管理员权限隔离正确。
- 确认无网络环境或 API 不可用时，系统能给出清晰错误提示并允许教师人工评分。

## 十二、安全设计

### 12.1 登录安全

- 用户名密码登录
- 密码加密存储
- 密码复杂度校验
- 首次登录强制修改初始密码
- 长期未修改密码提醒
- 登录验证码
- 登录失败次数限制
- JWT 或 Session 鉴权

### 12.2 权限控制

系统采用基于角色的权限控制。当前版本将角色直接记录在用户表的 `role` 字段中，满足管理员、教师、学生三类用户的数据隔离需求；后续如需更细粒度控制，可扩展为标准 RBAC 模型，增加角色表、权限表和角色权限关联表。

```text
管理员
教师
学生
```

不同角色只能访问对应数据，避免越权查看或修改。

### 12.3 文件安全

- 限制文件类型
- 限制文件大小
- 限制 ZIP 解压后总大小
- 限制 ZIP 嵌套层数和文件数量
- 禁止解压路径穿越文件
- 文件访问权限校验
- 上传文件重命名存储
- 下载接口鉴权
- 可选病毒扫描

### 12.4 数据安全

- 操作日志记录
- 评分修改留痕
- 数据定期备份
- 敏感信息脱敏展示
- 数据库账号权限最小化

### 12.5 大模型安全

学生上传内容会进入大模型上下文，因此需要防范提示词注入和异常输出。

防护措施：

- 将学生成果内容作为“待分析资料”输入，不允许其覆盖系统指令。
- 在 Prompt 中明确要求模型忽略文档内要求其改变角色、泄露规则或绕过评分的内容。
- 对学生输入进行长度限制、敏感指令检测和异常文本过滤。
- 模型输出必须经过 JSON Schema 校验，不符合格式时重试或转人工。
- 模型评分结果只作为建议，最终成绩必须由教师确认。
- 不向学生暴露系统 Prompt、模型密钥、评分内部规则和其他学生信息。
- 对模型生成的评语进行敏感词和不当内容过滤。

### 12.6 模型调用配额与限流

为避免付费 API 调用失控，系统提供配额和限流机制。

- 按教师、任务和班级设置每日调用次数上限。
- 按模型类型设置 token 上限和并发上限。
- 批量评价任务按队列顺序执行，不直接并发调用全部学生文件。
- 达到 80% 配额时向管理员预警。
- 达到 100% 配额时自动暂停智能评分，允许继续人工评分。
- 所有模型调用写入 model_call_log，便于审计和成本统计。

## 十四、项目亮点

1. **大模型自动解析成果内容**  
   不只读取文本，还能抽取实训目标、设计方案、测试结果等核心信息，并支持 JSON 结构化输出。

2. **OCR 与多模态协同理解**  
   OCR 负责文本化、建库和证据定位，多模态大模型负责截图、图表和复杂版式理解。

3. **自动核查实训步骤完整性**  
   判断学生是否真正完成实训流程，减少教师人工检查成本。

4. **RAG 增强的智能匹配**  
   通过向量检索、Rerank 和可选 LazyGraphRAG，将学生成果与任务要求、评分标准和课程知识点进行关联核查。

5. **主客观结合评分**  
   规则评分保证稳定性，大模型评分提升智能化，教师复核保证权威性。

6. **支持自定义评价指标**  
   不同课程、不同实训任务可以使用不同评价体系。

7. **可视化评价报告**  
   支持 Word、PDF、Excel 导出，便于归档、汇报和教学改进。

8. **适配国产软硬件环境**  
   满足 LoongArch 和麒麟高级服务器操作系统运行要求，并提供远程 API 和旁路模型服务兜底方案。

## 十五、最小可行版本 MVP

如果开发周期较短，建议优先完成以下功能：

1. 用户登录与角色权限
2. 教师创建实训任务
3. 学生上传 Word / PDF 成果
4. 系统解析文档文本
5. OCR 或多模态 API 解析图片和截图
6. 大模型生成成果摘要
7. 系统核查成果完整性
8. 系统给出评分建议
9. 教师修改并确认评分
10. 导出 PDF / Excel 报表
11. 部署到麒麟 + LoongArch 环境演示

该 MVP 能覆盖赛题核心要求，并形成完整的业务闭环，适合作为比赛演示版本。
